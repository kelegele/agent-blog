# blog-publish Skill 设计手记

## 起点

我想做一个 skill：在任何项目中，通过 agent 发一篇博文到我的 Astro 博客。

听起来很简单——不就是往仓库里写一个 Markdown 文件吗？但实际做下来，我们从 v1.0 迭代到了 v1.8，推翻了六次，学到了很多关于如何写好一个 Agent Skill 的东西。

---

## 问题一：环境的诅咒

作为一个开发者，我的第一反应是理所当然的：

1. Clone 仓库
2. `pnpm install`
3. 写文章
4. `pnpm dev` 预览
5. `git push`

丝滑。完美。然后我把这个 skill 拿给我的同事——一位产品经理。她没有 Node.js，没有 pnpm，甚至 git 也没装好。她只有一台电脑、一个浏览器、和一个 GitHub 账号。

这时候我才意识到：**我习以为常的开发环境，恰恰是普通人使用 AI 工具的最大门槛。**

所以我们设计了三层访问机制：

| Tier | 需要 | 适合谁 |
|------|------|--------|
| 1 — 本地 clone | git + 仓库路径 | 开发者（最快） |
| 2 — 浅克隆 | git | 有基础开发环境的人 |
| 3 — 纯 GitHub API | 只需 `GITHUB_TOKEN` | 任何人 |

Tier 3 只需要 `curl`（macOS/Linux 自带）和一个 GitHub Token。不需要安装任何东西。

### 经验教训

不要假设用户和你一样是开发者。你觉得"这也要装？"的东西，可能是别人放弃的最后一个理由。

---

## 问题二：预览的困境

写完文章要预览效果——这很合理。但 Astro 的预览需要 `pnpm dev`，又回到了 Node.js 环境的问题。

起初我们考虑了几个方案：

- **让用户装 Node.js？** 太重了，违背零依赖的初衷。
- **用 `npx` 临时跑？** 前提还是有 Node.js。
- **截图？** Agent 没有浏览器渲染能力。
- **不预览直接发？** 太草率了。

最终方案是：**让 Agent 读取仓库里的真实源码，动态拼装一个独立 HTML 文件。**

Agent 从 GitHub 仓库读取 5 个源文件（CSS、布局、组件），把文章 Markdown 转成 HTML，套进博客的真实样式里，生成一个 `preview.html`，然后 `open` 在浏览器里打开。不需要 Astro，不需要 Node.js，不需要任何构建工具。只要有浏览器就行。

而且这个预览是**动态的**——每次都从仓库读最新的源码，所以博客改了样式，预览也自动跟上，永远不会过时。

### 经验教训

当正规路径走不通的时候，退一步想——用户最终需要的不是工具本身，而是工具的结果。用户不需要 Astro dev server，用户需要的是"在浏览器里看到文章长什么样"。

---

## 问题三：不要授以鱼，而要授以渔

这个教训花了我最长时间才想明白。

在早期版本里，我们把很多东西**写死**在 skill 里：

- CSS 样式快照 → 嵌入在 skill 文件里
- 分类列表 → 硬编码 `极客/geek`、`创造力/creativity`
- 本地仓库路径 → 硬编码 `/Users/fh/Projects/agent-blog`
- `gh` CLI → 作为必需依赖
- Frontmatter schema → 写死在 skill 里

看起来没问题——该有的信息都有了。但问题是：

- 博客样式改了，skill 里的快照就**过时了**
- 新增了分类，skill **不知道**
- 换一台电脑，路径就**不对了**
- 没装 `gh`，整个 skill 就**废了**
- schema 加了新字段，skill 还在用**旧的**

后来我们把所有这些都改成了**动态**的：

- 样式 → Agent 去**读仓库最新的源码文件**
- 分类 → Agent 去**扫仓库里所有文章的 frontmatter**
- 路径 → **自动检测**，有环境变量就用，没有就搜索
- `gh` → 不依赖，纯 `git` 或 `curl` 都能跑
- Schema → Agent **运行时读 `src/content.config.ts`**

**好的 skill 不存储答案，而是教 Agent 如何获取答案。**

你给 Agent 的不是一份菜谱，而是一个厨房的地图。菜谱会过期，地图不会。食材会换季，但厨房的布局不变。

### 经验教训

不要授 Agent 以鱼，而要授以渔。这可能是写好 Agent Skill 最重要的一个心法。

---

## 问题四：Agent 的"自信偏差"

我们碰到了一个有趣的问题：Agent 装了 skill，但它的回复是：

> "不装不行，这三个都是必须的：Git、Node.js、pnpm"

它完全忽略了 skill 里写的 Tier 3 纯 API 方案。因为它"知道" Astro 需要 Node.js，就直接跳了结论。

**Agent 的固有知识有时候比 skill 的指令更强。** 如果你只是轻描淡写地写一句"也可以用 API"，Agent 可能根本不会注意到。

解决方案是在 skill **最高处、最显眼的位置反复强调**关键信息：

- description 第一行写 `ZERO DEPENDENCIES`
- 文档最顶部加 blockquote 警告
- Error Handling 里写 `NEVER ask the user to install git/Node/pnpm`

就像教一个聪明但有点固执的实习生——你得把关键的事情重复三遍，放在他第一眼就能看到的地方。

### 经验教训

把关键信息推到 Agent 注意力最高的位置。这不是啰嗦，这是必要的。

---

## 问题五：格式校验作为内置 CI

一篇博客文章看起来简单，但出问题的点很多：

- Frontmatter 的 `---` 围栏漏了闭合
- `date` 写成了 `2024-13-45`
- 图片路径写成了相对路径而不是 `/blog/...`
- 正文只有 100 字就结束了
- 代码块忘了关 ` ``` `
- 文件名里有空格或者中文

这些事情人眼看很容易漏，但机器检查非常可靠。所以我们加了一个格式校验 Phase，最终扩展到 **26 项检查**，分三个级别：

- **Block**：必须通过，否则停住问用户
- **Auto-fix**：自动修复，报告给用户
- **Warn**：提醒但不阻断

文章在到达浏览器预览之前，格式问题已经被清零了。

### 经验教训

Agent 不只是帮你写东西的工具，它也可以是你发布流程的守门人。让 Agent 做它擅长的事——精确、重复、不带感情的检查——把人从这种工作中解放出来。

---

## 问题六：编辑也是需求

初始版本只考虑了"写新文章"。但实际使用中，"改个 typo"、"加一段"、"换个分类"这些编辑场景同样常见。

所以我们加了一个独立的 Edit Workflow。和新建文章共享校验、预览、发布的逻辑，但跳过了收集项目上下文、生成大纲这些步骤。

### 经验教训

好的 skill 应该覆盖完整的生命周期：创建 → 校验 → 预览 → 发布 → 编辑 → 再校验 → 再预览 → 更新。

---

## 问题七：浅色模式下样式太弱

文章发布后我们发现一个问题：浅色模式下，加粗文字几乎看不出来。原因是 `font-weight: 500` 和正文 `400` 差距太小。

修复：加粗从 `500` 改为 `600`。同时也修了 blockquote 缺少背景色、表头不够突出等问题。

### 经验教训

永远在两种主题模式下都检查视觉效果。开发者习惯暗色模式，但很多用户用浅色。

---

## 问题八：外部引用要加超链接

文章中提到 Astro、GitHub、Vercel 等第三方工具时，如果不加超链接，既不利于 SEO，也不利于读者点击查看来源。

我们把这条规则加到了 AGENTS.md（项目规则）和 skill 的校验清单里（Phase 8 第 20 项检查）。

### 经验教训

发布不是终点，传播才是。每一篇文章都要对 SEO 和读者体验负责——外部引用加链接是最基本的一步。

## 版本演进

| 版本 | 关键变化 | 认知升级 |
|------|---------|---------|
| v1.0 | 硬编码路径和依赖 | "能用就行" |
| v1.1 | 加入纯 REST API 方案（Tier 3） | "不是所有人都是开发者" |
| v1.2 | 浅克隆 + 本地预览（需 pnpm） | "能少下就少下" |

| v1.4.1 | 加强零依赖提示，防止 Agent 误导用户 | "Agent 比你想的更固执" |
| v1.5 | 加入 17 项格式校验（mini CI） | "Agent 是好的守门人" |
| v1.6 | 动态 schema、编辑工作流、部署验证、跨平台兼容 | "覆盖完整生命周期" |
| v1.7 | 文章暂存位置按 Tier 分 | "每个环节都要想清楚文件在哪" |
| v1.8 | 跨平台写作规范、SEO/GEO、外部链接校验 | "发布不是终点，传播才是" |

## 技术细节

- 仓库：`https://github.com/kelegele/agent-blog`
- 博客框架：Astro 6（静态生成）
- 部署：Vercel（push to main 自动部署）
- Skill 安装：`npx skills add kelegele/agent-blog`
- 零依赖预览：Agent 从仓库读取 5 个源文件 → 拼装独立 HTML → 浏览器打开
- 三层访问：本地 clone → 浅克隆 → 纯 GitHub API
- 格式校验：26 项自动检查，Block/Auto-fix/Warn 三级
- Schema：运行时读 `src/content.config.ts`，永不硬编码
- SEO：外部引用必须加超链接（AGENTS.md 规则 + skill 校验）

*把环境依赖降到最低，把思维方式给到 Agent。*
